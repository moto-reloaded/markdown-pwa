# MD Atelier 開発ロードマップ

このファイルは、MD Atelier の開発段階をフェーズごとに整理したものです。  
目的は、Markdown を知らない人でも自然に記事を書ける PWA として、機能追加の順番を迷わないようにすることです。

## 現在地

- 現在バージョン: v1.0.0
- 今日の対象: Phase 1.0 から Phase 1.2
- 次の候補: Phase 1.3 音声入力

## 基本方針

- Markdown を直接知っている人にも、知らない人にも使いやすくする。
- ボタン名やUIは、Markdown記法名ではなく、Microsoft Word / Office 系のような「見た目ベース」の表現に寄せる。
- `標準 Markdown` と `note.com` の違いは、ユーザーに別ボタンとして見せすぎず、モード選択で吸収する。
- Chrome / Edge では直接保存や音声入力などの強い機能を使い、Safari / Firefox ではダウンロード保存などの安全な代替動作へ切り替える。
- スマホでも「編集」と「表示」を行き来しやすくする。

## Phase 1.1: 編集UIの整理

現在の `note大` / `note小` のようなボタン表記をやめ、全体をOffice系の編集UIに寄せます。

進捗: 実装済み

### 方針

ボタンは見た目ベースにします。

- 太字
- 斜体
- 取り消し線
- 大見出し
- 中見出し
- 小見出し
- 箇条書き
- 番号付きリスト
- チェックリスト
- 引用
- リンク
- 画像
- 表
- コード
- 区切り線
- 数式
- 埋め込み

`標準 Markdown` / `note.com` の違いは、ボタンを分けるのではなく、書式モードで切り替えます。

### 実装イメージ

```text
[書式: 標準 Markdown ▼]

太字  斜体  取消線 | 大見出し  中見出し  小見出し | 箇条書き  番号 | 引用  リンク  画像 | 表  コード  数式
```

`note.com` モードでは、同じ `大見出し` ボタンでも note に合わせた Markdown を出力します。

例:

| 操作 | 標準 Markdown | note.com |
| --- | --- | --- |
| 大見出し | `# 見出し` | `## 見出し` |
| 中見出し | `## 見出し` | `### 見出し` |
| 太字 | `**文字**` | `__文字__` |
| 引用 | `> 引用` | `> 引用` |
| 数式 | 任意 | `$${...}$$` |

### 完了条件

- note専用ラベルがツールバーから消える。
- ユーザーは書式モードを選ぶだけで、同じボタンを使える。
- Playwright で標準モード / note.com モードの出力差を確認できる。

## Phase 1.3: 音声入力

Chrome / Edge の Web Speech API を使って、カーソル位置に音声入力できるようにします。

### 方針

- `SpeechRecognition` / `webkitSpeechRecognition` を機能検出する。
- 非対応ブラウザではマイクボタンを非表示または無効化する。
- 日本語 / 英語の認識言語を選べるようにする。
- 認識結果は現在のカーソル位置に挿入する。
- README に、Chrome ではサーバーベース認識が使われる場合があることを明記する。

### 注意点

Chrome の音声認識は、ブラウザ標準機能ではあるものの、環境によってサーバーベース認識になる場合があります。完全なローカル処理として説明しないようにします。

### 完了条件

- Chromeでマイク許可後、文章が挿入される。
- 非対応ブラウザで壊れない。
- 音声入力中の状態表示がある。

## Phase 1.4: プレビューから編集を助ける

HTML表示側を直接編集する前に、プレビューから Markdown 編集を助ける機能を入れます。

### 方針

プレビューを直接編集可能にすると、HTMLからMarkdownへ戻す変換が難しくなります。まずは「プレビューをクリックすると対応するMarkdown位置へ移動する」機能を優先します。

### やること

- プレビューの見出しや段落をクリックすると、編集欄の近い位置へ移動する。
- 見出し一覧を表示できるようにする。
- 目次から該当見出しへジャンプできるようにする。

### 完了条件

- 長い文書でも目的の編集箇所へ移動しやすい。
- Markdown本文は壊さない。

## Phase 2.0: WYSIWYG編集の検討

プレビュー側で直接編集する本格機能は、Phase 2.0 として検討します。

### 検討項目

- 段落や見出しだけ編集できる簡易WYSIWYGにするか。
- HTML編集結果をMarkdownに戻す変換ルールをどうするか。
- 表、リンク、画像、note拡張、数式をどう扱うか。
- Markdown編集とWYSIWYG編集のどちらを正とするか。

### 判断基準

- Markdownが壊れない。
- note.comへの貼り付け結果が安定する。
- ユーザーが「どちらを編集しているのか」で迷わない。
